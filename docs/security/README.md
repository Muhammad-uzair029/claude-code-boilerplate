# `docs/security/` — Security Documentation Index

Everything security-related lives here: audit history, remediation history, and the open punch list. New security work goes in this folder; nothing security-related goes elsewhere.

---

## Quick start for a new developer

You just joined the team and were pointed at security work. Read these, in this order:

1. **[open-items.md](./open-items.md)** — the only file that says "what's still open right now". Has priority buckets (P0 = this week, P1 = this month, P2 = this quarter). Pick a P0 if nothing's claimed yet.
2. The **most recent** `security-remediation-YYYY-MM-DD.md` in this folder — tells you what state the codebase was in last time someone touched the security surface. Sometimes the audit script false-negatives on fixes already on disk (e.g. due to scan-root config mismatch); the remediation file is how you know what's actually there.
3. The skill spec: [`.claude/skills/security-audit/SKILL.md`](../../.claude/skills/security-audit/SKILL.md) — explains the 42 rules, the four modes (`/security-audit`, `/security-audit <N>`, `/security-audit api-mask`, `/security-audit report`), and what auto-fix can and can't touch.

You don't need to read every historical remediation file. The most recent one + `open-items.md` is the current picture.

## Scope rule — what goes where (configurable per team)

Many teams find it useful to split security work into two streams that have different blockers, reviewers, and cadence:

| Stream | Lives in | Examples |
|---|---|---|
| **Third-party / account / external-service work** — AWS console, Cloudflare dashboard, GitHub Actions, IAM, Secrets Manager, CI scanner choice, office IP allowlist | **`open-items.md`** (this folder) | "Switch from S3 access keys to AssumeRole", "Put origin behind Cloudflare", "Enable GuardDuty" |
| **Pure-code work** — code, config, dependency upgrades, tests | The relevant **feature branch** and the **"Still open" section of the matching `security-remediation-*.md`** | "Add CSRF middleware", "PII redact in logs", "Write integration tests for CORS allowlist" |

**This split is optional but recommended.** When the team adopts it, the rule is: if "I can fix this by typing in my editor and pushing", it's code-stream; if "I need to log into a dashboard or talk to someone with admin", it's `open-items.md`.

If your team hasn't adopted the split, `open-items.md` holds everything that's open, code or not. Either is fine — just be consistent. Document the choice here once you make it.

## File types

| File pattern | What it is | Mutability | Who writes it |
|---|---|---|---|
| `open-items.md` | **Single source of truth for what's still open.** Prioritized (P0–P2). Items retire to the **Done** table at the bottom. | Mutable — updated after every remediation pass | Whoever finishes a remediation pass |
| `security-remediation-YYYY-MM-DD.md` | **Snapshot of one remediation pass.** What the audit found, what was fixed, the still-open code items, verification commands and outputs, files changed. | **Immutable once committed** — one file per pass; never append | Whoever ran the remediation that day |
| `api-mask-audit-YYYY-MM-DD.md` | **Output of Mode 3 — `/security-audit api-mask`.** Per-endpoint API response over-sharing assessment, scored HIGH / MEDIUM / LOW / CLEAN. | **Immutable once committed** — one file per audit run | Whoever ran the audit that day |
| `pen-test-report-YYYY-MM-DD.md` | **Auto-generated by `.claude/skills/security-audit/scripts/pen-test-report.sh`.** Full 42-rule audit with OWASP mapping. | **Immutable** | The skill's report generator |
| `README.md` (this file) | Workflow + conventions + index. | Mutable — kept current with the workflow | Whoever changes the workflow |

If two passes of the same type land on the same UTC day, suffix the filename: `-am`, `-pm`, `-2`. Never overwrite a prior snapshot.

## Workflow when you run the audit and apply fixes

```
1. Read open-items.md             ← know what's already tracked
2. Read most recent remediation   ← know what's already in the code
3. Run audit                      ← bash .claude/skills/security-audit/scripts/run-all.sh
4. For each finding, route it:
     - Third-party / account work → goes in open-items.md
     - Pure-code work             → goes in your branch + remediation file
5. For findings already tracked:
     - if priority or blast radius changed, edit the entry in place
     - if you're fixing it today, note it (you'll close it in step 8 or 9)
6. For findings NOT tracked yet:
     - add to open-items.md (correct priority bucket) OR
     - put in your branch + flag in the remediation "still open" list
7. Apply auto-fix-eligible items (per SKILL.md "Auto-fix Eligibility")
8. Move every fixed item to the Done table at the bottom of open-items.md
9. Write a new security-remediation-YYYY-MM-DD.md describing the day's work,
   including the still-open code items
10. Re-run the affected rules to confirm the fix landed
    (be aware of the scan-root blindspot — read code, not just script output)
11. Mask the new doc — bash .claude/skills/security-audit/scripts/mask-findings.sh <file>
```

## "What's still open?" — fast path

If someone asks "what's left to do on security?" — **do not re-run the audit first**. The answer is:

1. **Third-party / account stream:** `open-items.md`, top to bottom, P0 → P1 → P2.
2. **Code stream:** "Still open" section of the **most recent** `security-remediation-*.md`.

Only re-run the audit if you suspect the open-items file is stale (e.g. nobody's updated it in a while, or you just merged a security PR).

## Severity definitions (used in remediation reports)

- **Critical** — pre-auth RCE, mass PII exfil, full takeover. Drop everything.
- **High** — auth bypass, individual-user PII exfil, dep CVE with a known exploit. This-week fix.
- **Moderate** — info disclosure, missing defense-in-depth that an attacker would chain. This-month fix.
- **Low** — hygiene, defense-in-depth that doesn't change exposure. This-quarter fix.

## Priority key (used in `open-items.md`)

- **P0** — do this week. Items in this bucket are typically Critical/High in severity and / or block the next deploy.
- **P1** — do this month. Real risk, but no active exploit path.
- **P2** — do this quarter. Hygiene, defense-in-depth, detection gaps.

Severity (what an attacker could do) and priority (when you should do it) usually agree, but not always. A Moderate finding behind a Cloudflare-only network can be P2; a Low finding that gates a compliance audit can be P0.

## Conventions

- **Never delete files in this folder.** Move-to-Done is the only retirement path for backlog items. Snapshots stay forever — they're the audit trail.
- **Never commit secrets here.** This folder is checked into git. Use placeholders like `<BUCKET>`, `<ACCOUNT_ID>`, `<office-CIDR>` instead.
- **Always date your files in UTC.** Helps with audit trails when someone else picks up the work.
- **One remediation pass = one `security-remediation-*.md` file.** Don't append to old ones; create a new one.
- **Respect the scope split (if adopted).** Third-party work → `open-items.md`. Pure-code work → branches + remediation files.
- **Read before writing.** Always read `open-items.md` and the most recent remediation file before reporting findings, to avoid double-counting items the team already knows about.
- **Mask before persisting.** Run `bash .claude/skills/security-audit/scripts/mask-findings.sh <file>` on any new doc before committing.
